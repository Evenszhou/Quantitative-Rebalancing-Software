## 📋 项目概览

### 团队情况
- **开发团队**: 1人（研究员兼开发）+ AI助手
- **投研流程**: 研究员/交易员讨论 → 投资委员会上会 → 决策执行
- **当前用户**: 内部使用
- **未来规划**: 可能商业化

### 技术栈
- **核心**: Python 3.10+, Streamlit, Pandas, NumPy, SciPy
- **数据存储**: JSON/YAML配置文件（避免过度设计）
- **实盘对接**: 迅投模拟盘 → miniqmt
- **资产规模**: 1000万模拟盘

### 项目约束
⚠️ **1人开发** - 必须极度简化，避免过度工程化
⚠️ **时间有限** - 优先做高ROI功能
⚠️ **可能商业化** - 架构要留有余地，但不能过度设计
⚠️ **支撑投研流程** - 软件是工具，不是替代人

---

## 🎯 开发策略：最小可行产品路线

### 核心原则
1. **支撑投研流程** > 追求学术前沿
2. **可解释性** > 黑箱算法
3. **实用稳定** > 功能数量
4. **快速迭代** > 完美架构
5. **业界验证的方法** > 最新酷炫方法

### 不做的事情
❌ 机器学习/深度学习方法（黑箱、过拟合、难解释）
❌ 复杂工程化（K8s、微服务、前后端分离）
❌ 17种方法全做（维护成本高）
❌ 过度抽象和设计模式

## 🔧 技术决策记录

### 为什么选择Streamlit而不是FastAPI+React？
- 1人开发，维护两套代码成本太高
- Streamlit够用，内部工具不需要太好的UI
- 快速迭代，改动方便

### 为什么不用PostgreSQL而是JSON文件？
- 数据量不大（观点数量有限）
- 简单，便于调试和版本控制
- 需要时随时可以迁移到数据库

### 为什么先做Black-Litterman？
- 直接支撑投研流程（研究员有观点）
- 可解释性强（IC能看懂）
- 业界标准，不是黑箱
- 商业化时的卖点

### 为什么不做机器学习方法？
- 过拟合风险极高
- 黑箱特性，无法向IC解释
- 样本外泛化差
- 维护成本高
